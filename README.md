# ODCA Document Log Web Parts

A suite of SharePoint Framework (SPFx) web parts for the Office of the Division Clerk of Court and Adjudication Support (ODCA), Supreme Court of the Philippines. These web parts provide document logging, status tracking, and lifecycle timeline visualization against a SharePoint Online list.

## Web Parts

### 1. Document Log

Creates a new document log entry in SharePoint with a multi-step form workflow:

1. **Details** — fill in document title, type (RCM/MISC), format (HC/SC), sender, status, optional remarks, and file attachments (browse or drag-and-drop)
2. **Review** — verify all fields before submission
3. **Log Code** — submits to SharePoint, uploads attachments, and polls for a reference code generated by Power Automate

Includes a live timer tracking time spent on each step, a printable routing slip, and clipboard copy for the generated reference code.

### 2. Document Status Update

Searches for an existing document by reference code and updates its status:

1. **Search** — enter a reference code (e.g. `RCM-SC-0001`) to find the document
2. **Detail** — read-only summary of the document's current state
3. **Update** — change the status and/or remarks, then submit via PATCH (`X-HTTP-Method: MERGE`)
4. **Success** — confirmation showing previous and new status

### 3. Document Timeline

Displays the full status change history of a document as a visual timeline:

1. **Search** — enter a reference code to look up
2. **Summary + Timeline** — queries both the Document Log Tracking list and the Document Status Log list, then renders:
   - Document summary card (title, type, format, from, current status, date logged)
   - Vertical timeline with each status transition (from/to status, duration, changed by, timestamp)
   - Current status shown with a pulsing indicator
   - Total elapsed time across all transitions

## SharePoint Lists

### Document Log Tracking

Primary list for document entries.

| Column | Internal Name | Type |
|--------|--------------|------|
| Title | Title | Text (document title) |
| Reference Code | ReferenceCode | Text (generated by Power Automate) |
| Document Type | Document_x0020_Type | Choice (RCM, MISC) |
| Document Format | Document_x0020_Format | Choice (HC, SC) |
| From | From | Text |
| Status | Status | Choice (Received, In Progress, For DCA Approval and Signature, Released, Filed) |
| Other Remarks | OtherRemarks | Text |

### Document Status Log

Audit log of status transitions, populated by Power Automate.

| Column | Internal Name | Type |
|--------|--------------|------|
| Title | Title | Text (reference code) |
| From Status | FromStatus | Text |
| To Status | ToStatus | Text |
| Duration | DurationHours | Number (minutes, decimal) |
| Changed By | ChangedBy | Text |
| Document Title | DocumentTitle | Text |

## Prerequisites

- Node.js >=18.17.1 <19.0.0
- SharePoint Framework v1.20.0
- A SharePoint Online site with the two lists above created and configured
- A Power Automate flow that generates the `ReferenceCode` on the Document Log Tracking list and logs transitions to the Document Status Log list

## Installation

```bash
git clone <repository-url>
cd document-log-webpart
npm install
```

## Local Development

```bash
gulp serve
```

This launches the SharePoint Workbench where you can add and test the web parts. You'll need to configure the site URL in each web part's property pane to point to a SharePoint site with the required lists.

In some cases, you need to install a developer trust certificate.

```bash
gulp trust-dev-cert
```

## Build and Package

```bash
# Production build
gulp bundle --ship

# Create the .sppkg package
gulp package-solution --ship
```

The package is output to `sharepoint/solution/document-log-webpart.sppkg`.

## Deployment to SharePoint

1. **Upload the package** — go to your SharePoint App Catalog (`https://<tenant>.sharepoint.com/sites/appcatalog`) > Apps for SharePoint > Upload `document-log-webpart.sppkg`
2. **Trust the package** — when prompted, check "Make this solution available to all sites in the organization" if you want tenant-wide deployment, then click **Deploy**
3. **Add to a site** — navigate to the target site > Site Contents > New > App > find "document-log-webpart-client-side-solution" and add it
4. **Add web parts to a page** — edit a SharePoint page, click **+** to add a web part, and search for:
   - "Document Log" (alias: `document-log`)
   - "Document Status Update" (alias: `DocumentStatusUpdateWebPart`)
   - "Document Timeline" (alias: `DocumentTimelineWebPart`)
5. **Configure** — open each web part's property pane to set the site URL and list names if they differ from the defaults

## Configuration

All three web parts support the same property pane pattern:

| Property | Default | Description |
|----------|---------|-------------|
| Site URL | Current site | SharePoint site URL containing the lists |
| List Name | `Document Log Tracking` | Name of the primary document log list |
| Status Log List Name* | `Document Status Log` | Name of the status transition log list |

*Status Log List Name is only available on the Document Timeline web part.

## Technical Notes

- All web parts use plain TypeScript with direct DOM manipulation (no React or other UI framework)
- SharePoint REST API calls use `SPHttpClient` from `@microsoft/sp-http`
- CSS is inlined in each web part's `render()` method with prefixed class names (`dl-`, `dsu-`, `dtl-`) to avoid collisions
- The Document Log web part includes `dispose()` cleanup for interval timers to prevent memory leaks
